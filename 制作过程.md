## 三、关键决策与为什么

- **为什么选 PySide6 而非 Tauri**：Tauri 需装 Rust 工具链，对 2-3 秒彩蛋收益低。PySide6 + PyInstaller 已足够漂亮，部署最简。
- **为什么分批释放金币而不是一次性**：用户反馈"基本都是同一层一起落下来"，调整为 8 批 × 350ms 间隔（总生成时间 2.5s），配合放缓的下落速度，得到"前后层次感"。
- **为什么最终只播一个长瀑布音而不是每枚落地各响一次**：30 - 50 枚同时叮当会变噪音；代码合成一个 4 秒瀑布音，一次播放，听感自然。
- **为什么弃用 QSoundEffect、改用 winsound**：用户反馈 exe 打包后没声音。PyInstaller `--onefile` 不稳定地打包 Qt 多媒体插件（`qtaudio_windows.dll` 等），QSoundEffect 运行时找不到插件就静默失败。`winsound` 是 Python 标准库调用 Windows 原生 `PlaySound` API，无任何 Qt 依赖，打包后必然可用。
- **为什么音效准备了"网络下载 + 代码合成"双路径**：pixabay 直链经常失效，代码合成做兜底。最终采用代码合成方案。
- **为什么不写单元测试**：核心价值（透明、穿透、下落视觉、系统调度）无法被单测可靠覆盖，改用手动 checklist。
- **为什么不加日志 / 托盘 / 配置 UI**：YAGNI，一次性彩蛋不值得这些工程。
- **为什么 uninstall 脚本也卸 CoinRainTest**：实施阶段额外创建了 10 分钟后触发的一次性测试任务，为了保持"一个脚本清理干净所有痕迹"的整洁，把它并入卸载列表。

## 五、遇到的问题与解决

### 问题 1：首次运行金币"同一层"一起落下

- **症状**：用户反馈"基本都是同一层一起落下来了，最好还是有点层次感"。
- **原因**：初始参数 BATCH_COUNT=3、BATCH_INTERVAL_MS=100，总生成时间仅 200ms，远小于下落时间，所以视觉上所有金币几乎同时出现。
- **解决**：BATCH_COUNT → 8，BATCH_INTERVAL_MS → 350（总生成 2.45s），与下落时间相当，形成持续倾泻感。同时放缓重力和初速度，让金币"飘"得更久。

### 问题 2：打包后的 exe 没声音

- **症状**：python 直接运行有声，PyInstaller 打包后的 exe 无声。
- **原因**：QSoundEffect 依赖 `Qt6Multimedia.dll` 和 `qtaudio_windows.dll` 等插件，PyInstaller `--onefile` 的自动 hook 未能完整打包这些多媒体插件。
- **解决**：弃用 QSoundEffect，改用 `winsound.PlaySound(path, SND_FILENAME | SND_ASYNC)`。winsound 是 Python 标准库直接调用 Windows 内核 API，无任何外部依赖，打包后必然可用。代码量从 5 行减到 2 行，反而更简洁。

### 问题 3（用户反馈）：单次叮声不够有"金币落地感"

- **症状**：用户反馈"声音不应该只是在最开始的时候响一下，应该是伴随着金币下落有哗啦啦的声音"。
- **原因**：原始音效是 0.35s 的单次叮，与视觉的 4.5s 持续下落不匹配。
- **解决**：重写音效生成脚本，在 4 秒时间轴上按 beta 分布（中间密集、两端稀疏）撒 28 个随机音高、随机时长的叮，混合归一化。播放时还是一次 PlaySound，但听感是持续"哗啦啦"瀑布声。

# 第二轮迭代 · 配置 UI + 到账大字 + 多次触发

- **日期**：2026-04-16
- **耗时**：约 3 小时（含设计、可视化 mockup 迭代、5 款金币 SVG 设计、编码、打包）
- **触发**：用户提出 3 项改进 → 对外发布做准备、到账大字、首次配置 UI

## 一、动机

第一版是个人玩具：参数硬编码、PowerShell 脚本注册、无界面、无到账数字。要"准备对外发布"，得让随便一个 Windows 用户下载 exe 双击就能配置自己的"每日收入"和触发节奏，并且每次触发能看到"今日到账 ¥X"的喜悦感。

## 二、需求三件套

1. **首次使用的图形配置界面**：双击 exe 弹出向导（每日收入 / 雨势档位 / 单次或多次 / 时间）；保存后任务计划自动注册；之后双击 exe 进管理界面（看摘要 / 启用停用 / 修改）
2. **金币雨中央叠加大字「今日到账 ¥XXX」**：count-up 0→目标，先 1 秒出字再落币
3. **多次触发模式**：填首次时间、末次时间、N 次数 → N 次等距分布，文案在「当前已到账」→「今日到账」之间累计闭合

## 三、设计过程的关键决策

| 问题 | 选项 | 选择 | 理由 |
|---|---|---|---|
| exe 分流 | A. 单 exe + CLI 参数 / B. 两个 exe | A | 一次构建、零维护成本 |
| 多次模式时间表达 | 时段+次数 / **首次+末次+次数** / 用户填 N 个 | 用户填首+末+N | 最精确，无歧义 |
| 多次到账金额 | **等额累计** / 等额每次 / 随机分配 | 等额累计 | 闭合自然，符合"已到账"语义 |
| 金币样式 | 单一固定 / 5 款备选 / 用户自填 | 5 款备选+混合 | 默认开元通宝，混合下落彩蛋 |
| 到账大字风格 | 编辑风 / **巨字金光** / 低调条幅 | 巨字金光 | 仪式感强，count-up 滚到目标 |
| UI 配色 | 暗色系 / **米白纸本+鎏金** / 国潮 | 米白+金 | editorial serif，避开常见暗色 UI |
| 任务计划多次模式 | 1 任务 N 触发器（XML）/ **N 个独立任务** | N 个独立任务 | 触发器无法绑不同 CLI 参数，独立任务最简 |
| 测试 | 0 单测（CLAUDE.md 原约定）/ **3+ 个纯函数 pytest** | 加 5 个 → 实际加了 15 个 | 多次模式的浮点取整、Config roundtrip 容易出 bug |

## 六、踩过的坑

- **Google Fonts 直链下载会拿到 HTML**（需要 JS）→ 改用 GitHub raw 直连 `raw.githubusercontent.com/.../master/...`
- **思源宋体变体字体 25 MB**：完整字符集太大，exe 从 44 MB 涨到 60 MB；遗留后续 subset 优化空间
- **schtasks 多触发器无法绑不同 CLI 参数**：放弃 XML 路线，改 N 个独立任务（`CoinRainDaily_1/2/3`）
- **schtasks 输出在中文 Windows 上是中文标题**："Scheduled Task State" 变成"任务状态"，"Next Run Time" 变成"下次运行时间" → `_single_task_status` 同时识别中英文
- **PySide6 QSS 动态 property（如 `[invalid="true"]`）**：必须 `setProperty + style.unpolish/polish` 才能让 QSS 重算
